Architecture & Tools

The journey of an ML model involves several key architectural decisions: deciding where the model comes from, choosing the best way to deploy it to users, and leveraging cloud services to automate and manage the entire lifecycle.

1. Understand Sources of ML Models

Before any MLOps pipeline begins, a decision must be made: are we building a new model, or using one that already exists?

Open Source Pre-trained Models

These models have already been trained on massive, public datasets (like Wikipedia text or millions of images). They are excellent for tasks that are common in the industry.

  • Benefit: Saves significant time and compute resources. They are often highly optimized and require minimal effort to get working.
  • Use Case: You use them directly or, more commonly, use **transfer learning**, where you fine-tune the pre-trained model on your smaller, specific dataset.
  • Examples: Models from Hugging Face for Natural Language Processing (NLP), or image classification models from TensorFlow Hub.

Training Custom Models

This involves building and training a model from scratch using your proprietary data. This is necessary when your problem is unique or when pre-trained models don't meet the required accuracy or performance.

  • Benefit: Highest potential accuracy for specialized tasks. Complete control over architecture and data usage.
  • Requirement: Demands extensive data collection, feature engineering, high computational power, and rigorous MLOps practices to manage complexity.

2. Methods to Use a Model in Production

Once a model is trained and validated, it needs to be accessible to other applications or users. The two main deployment methods balance control against operational effort.

Managed API Service (e.g., AWS SageMaker Endpoint)

This is the cloud-native approach. The model is uploaded to a platform that handles all the operational tasks automatically.

  • How it works: The cloud provider (like AWS) hosts the model, creates a dedicated REST API endpoint, and automatically manages scaling, load balancing, health checks, and infrastructure provisioning.
  • Pros: **Low operational overhead**, fast time-to-market, automatic scaling to handle varying traffic.
  • Cons: Less flexibility for highly customized deployment logic or unusual networking requirements.

Self-Hosted API (e.g., using EC2 or Kubernetes)

The user packages the model with a web framework (like Flask or FastAPI) and deploys it onto self-managed infrastructure (Virtual Machines or container orchestrators).

  • How it works: The organization controls the virtual machines (EC2) or Kubernetes cluster (EKS) where the model is run. They handle all the configuration and scaling.
  • Pros: **Maximum control** over the environment, networking, and security. Potentially cheaper for very high, consistent load.
  • Cons: **High operational burden**; requires dedicated DevOps/MLOps expertise to manage scaling, updates, and reliability.

3. Relevant AWS Services for the ML Pipeline

AWS offers **Amazon SageMaker** as its primary service for the end-to-end ML workflow, encompassing tools for nearly every stage of the pipeline.

Data Preparation and Feature Management

  • Amazon SageMaker Data Wrangler: A visual tool used for data exploration, aggregation, and cleaning (EDA and Pre-processing stages).
  • Amazon SageMaker Feature Store: A centralized repository for storing, updating, sharing, and serving machine learning features. Ensures the same features used during training are used for real-time inference, solving the **training-serving skew** problem.

Model Training, Tracking, and Deployment

  • Amazon SageMaker: The comprehensive platform used for launching **Training Jobs** (model training stage) and managing **Experiments** (experimentation stage).
  • Amazon SageMaker Endpoints: The service used for deploying models as a **Managed API Service** for real-time inference (deployment stage).

Monitoring and Governance

  • Amazon SageMaker Model Monitor: Automatically detects data drift and model quality drift in real-time on deployed models (model monitoring stage).
  • Amazon SageMaker Pipelines: Used to define and orchestrate repeatable, automated CI/CD workflows for retraining and deployment (repeatable processes stage).
